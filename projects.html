Twixt CLI

CS0.101 · SEMESTER 1 · PROF. GIRISH VARMA

WHAT IT DOES

A fully playable command-line version of Twixt — a two-player strategy board game where each player places pegs and links them to build a connected path across the board. The program handles move validation, player state, and board rendering in the terminal, updating the display after every turn without flicker. Win detection runs depth-first search on the graph of each player's connections to find a valid path from one side to the other.

WHY I BUILT IT

First semester project in CS0.101. It was the first time I applied graph algorithms to something that actually moved — seeing DFS run on a live board state made the theory feel real in a way that textbook problems don't quite manage.

WHAT I'D DO DIFFERENTLY

The board rendering was the most time-consuming part. Drawing the grid with raw - and | characters in a loop got messy fast — any layout change meant rewriting the whole rendering logic. I'd build a proper abstraction layer for the display from the start, separate from the game state, so they can change independently.

TECHNOLOGY TRADEOFFS

DFS is correct for win detection but re-runs on every move, which is fine for a 24×24 board. For a larger variant you'd want an incremental union-find approach instead. I also used a 2D array for the board — simpler than a graph adjacency list, but meant a lot of index arithmetic throughout the code.

Music Library Manager

CS0.101 · SEMESTER 1 · PROF. GIRISH VARMA

WHAT IT DOES

A persistent music library simulation that stores songs and playlists across sessions using CSV files. Supports full CRUD operations — adding, removing, updating, and searching songs. Playlists are stored as circular doubly linked lists for efficient forward and backward traversal, and all memory is managed manually with malloc and free.

WHY I BUILT IT

Another CS0.101 project, but this one landed at exactly the right moment — I'd just learned file handling, and suddenly here was a real use case for it. Reading and writing structured data to disk, parsing CSV records, managing memory across sessions. It clicked in a way it wouldn't have as a standalone exercise.

WHAT I'D DO DIFFERENTLY

The CSV parser was character-by-character and broke on edge cases — song names with commas in them would corrupt the parse entirely. I'd write a proper quoted-field parser from the start, or at least add a delimiter escape scheme. Also, circular doubly linked lists are great for traversal but O(n) for search — a hash map would be faster if the library grew large.

TECHNOLOGY TRADEOFFS

Manual memory management with malloc/free gave precise control but required tracking every allocation. One missed free and the library would leak memory with every session. The circular doubly linked list structure was good for playlist operations but overkill for a simple song index.

Smart Water Level Monitor

CS3.303 · SEMESTER 2 · PROF. KAVITA VEMURI & PROF. SURESH PURINI

WHAT IT DOES

An IoT system that monitors water level in a container using three sensors: an HC-SR04 ultrasonic sensor measures distance to the water surface, an analogue water level sensor provides a reference ADC reading, and an IR sensor detects whether the container is even present. The two main sensors are calibrated against each other using linear regression, and the processed data is transmitted over Wi-Fi via HTTP to ThingSpeak for remote monitoring.

WHY I BUILT IT

Lab mini-project for our IoT course. The sensor fusion aspect was genuinely interesting — the ultrasonic sensor alone gave noisy readings due to surface reflections, and combining it with the analogue reference made the system significantly more stable. Watching data appear on ThingSpeak in real time from a physical device was a different kind of satisfying from pure software work.

WHAT I'D DO DIFFERENTLY

The linear regression calibration was computed manually from 10 data points — not a lot. I'd automate the calibration process and take more samples. I'd also add a moving average filter on the ultrasonic readings to smooth out the reflection noise rather than relying on calibration to paper over it. For deployment beyond a lab, MQTT would replace HTTP — it's designed for unreliable networks and handles reconnections better than raw HTTP over Wi-Fi.

TECHNOLOGY TRADEOFFS

Using two sensors instead of one adds cost and wiring complexity, but the accuracy improvement was worth it in this case — the ultrasonic sensor alone had measurement errors of several centimetres near the water surface. The ESP32 over Wi-Fi is convenient for prototyping but power-hungry for a device that should ideally run for months on battery.
